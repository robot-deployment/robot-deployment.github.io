    Cyclone config

    CycloneDDS Config

    This chapter provides three working recipes for getting your turtlebot sending messages over ROS -- albeit with cycloneDDS and unicast.

    The downside of unicast is that, because these messages are sent between two peers, these networks can't scale to many machines, as the load on the network grows quite quickly.

    Below we describe the initial setup of your turtlebot, virtual machine, and create3 to prepare it for use with these recipes.

    Initial Setup

    First, you will need a wired ethernet adapter on your host PC. You can use a USB ethernet adapter if your computer doesn't have one.

    Set the ip address of your host machine to:

    ip: 192.168.5.5/24
    gateway: 192.168.5.1

    For your virtual machine, you will need to add a second bridged adapter to your setup. While the VM is off, go to settings -> network and go to adapter 2.

    Bridge to your ethernet adapter{height=200}

    Your guest virtual machine will need specific setups for each recipe, but you don't need to prepare it at this point.

    Raspberry Pi

    To prep the raspberry pi, you will want to disable the discovery service

    sudo systemctl stop discovery.service
sudo systemctl disable discovery.service

    You also need to disable the turtlebot 4 startup service, as it doesn't work right.

    sudo systemctl stop turtlebot4.service 
sudo systemctl disable turtlebot4.service

    You will also want to delete /etc/netplan/40-ethernets.yaml, as the following config files for /etc/netplan/50-wifis.yaml include your ethernet addresses.

    Create3 Setup

    At :8080 of your raspberry pi's ip address, go to application -> configuration

    Recipe 1: Without Wifi

    This first recipe tries to eliminate wifi altogether, in order to ensure high transmission speeds. Unfortunately, though, because the raspberry pi's two ethernet adapters can't be bridged (using settings I could find), this means the Create3 and VM guest cannot publish and subscribe to each other's topics. Though the raspberryi pi can see both sets of messages, it would need to be set up to act as a broker, retransmitting important messages from the create3 out to the VM and vice-versa. Thus, though this works, it is not a complete working example between all devices.

    Create3

    in addition to the initial setup described above, go to the beta features menu -> override rmw profile

    The field should be:

    <CycloneDDS>
    <Domain>
        <General>
            <DontRoute>true</DontRoute>
            <AllowMulticast>false</AllowMulticast>
            <EnableMulticastLoopback>true</EnableMulticastLoopback>
        </General>
        <Discovery>
          <ParticipantIndex>auto</ParticipantIndex>
          <MaxAutoParticipantIndex>100</MaxAutoParticipantIndex>
        </Discovery>
    </Domain>
</CycloneDDS>

    ~/cyclonedds.xml

    <CycloneDDS>
    <Domain>
        <General>
            <Interfaces>
                <NetworkInterface name="eth0" priority="default" 
                multicast="default"/>
                <NetworkInterface name="usb0" priority="default" 
                multicast="default"/>
            </Interfaces>
            <!--<DontRoute>true</DontRoute>-->
            <AllowMulticast>false</AllowMulticast>
            <EnableMulticastLoopback>true</EnableMulticastLoopback>
        </General>
        <Discovery>
          <ParticipantIndex>auto</ParticipantIndex>
          <MaxAutoParticipantIndex>100</MaxAutoParticipantIndex>
          <Peers>
            <!--The static ip address for your raspberry pi's eth0 adapter-->
            <Peer Address="192.168.185.1"/> 

            <!--The static ip address for your raspberry pi's usb0 adapter-->
            <Peer Address="192.168.186.1"/> 

            <!--The static ip address for your create3's usb0 adapter-->
            <Peer Address="192.168.186.2"/> 

            <!--The static ip address of your virtual ethernet adapter 
            connected to your host's ethernet adapter-->
            <Peer Address="192.168.185.6"/> 
          </Peers>
    </Discovery>
    </Domain>
</CycloneDDS>

    Recipe 2: Create3 connected to the VM over wifi

    This recipe addresses the fact that in recipe 1, the virtual machine and create3 are not on the same subnet and cannot directly communicate. This eliminates the need for the Raspberry pi to serve as a re-transmitter of important topics, but requires a cable still to the raspberry pi to ensure high-speed data transfers for the OakD Camera

    Create3

    1. The configuration for this recipe should be the same as the initial setup described at the beginning of this chapter. However, you should connect the wifi to ASPIN

      The Create3 cannot be set to use a static IP address, therefore, once it has been connected, note the IP address

    2. Next, in the beta features menu -> override rmw profile

    The field should be:

    <CycloneDDS>
    <Domain>
        <General>
            <Interfaces>
                <NetworkInterface name="wlan0" priority="default" 
                multicast="default"/>
                <NetworkInterface name="usb0" priority="default" 
                multicast="default"/>
            </Interfaces>
            <DontRoute>true</DontRoute>
            <AllowMulticast>false</AllowMulticast>
            <EnableMulticastLoopback>true</EnableMulticastLoopback>
        </General>
        <Discovery>
          <ParticipantIndex>auto</ParticipantIndex>
          <MaxAutoParticipantIndex>100</MaxAutoParticipantIndex>
          <Peers>
            <!--The static ip address for your raspberry pi's eth0 adapter-->
            <Peer Address="192.168.185.1"/> 
            <!--The static ip address for your raspberry pi's usb0 adapter-->
            <Peer Address="192.168.186.1"/> 
            <!--The static ip address for your create3's usb0 adapter-->
            <Peer Address="192.168.186.2"/> 
            <!--The static ip address of your virtual ethernet adapter 
            connected to your host's ethernet adapter -->
            <Peer Address="192.168.185.6"/> 
            <!--The static IP address of your Virtual Machine 
            (over the host wifi)-->
            <Peer Address="10.166.219.70"/> 
          </Peers>
        </Discovery>

    </Domain>
</CycloneDDS>

    Recipe 4: Ethernet only, with the Create3 Republisher (next chapter)

    This recipe tries to eliminates wifi altogether, in order to ensure high transmission speeds. This solves the bridging problem from Recipe 1, allowing any device on the raspberry pi's network adapters to gain access to the Create3's most important topics. This could be considered a replacement or fix to recipe 1

    Create3

    in addition to the initial setup described at the beginning of the chapter, go to the beta features menu -> override rmw profile

    The field should be:

    <CycloneDDS>
    <Domain>
        <General>
            <DontRoute>true</DontRoute>
            <AllowMulticast>false</AllowMulticast>
            <EnableMulticastLoopback>true</EnableMulticastLoopback>
        </General>
        <Discovery>
          <ParticipantIndex>auto</ParticipantIndex>
          <MaxAutoParticipantIndex>100</MaxAutoParticipantIndex>
        </Discovery>
    </Domain>
</CycloneDDS>

    Furthermore, since you are working completely over ethernet, you can disable the create3's wifi connection using "Beta Features" -> "Forget Wifi Network"

    Also, rename the namespace from "/c3_03" to "/rpi_03/_do_not_use" (replacing 03 with your turtlebot's namespace).

    You will also need to reconfigure the safety parameters according to the next chapter

    Save and restart the application.
